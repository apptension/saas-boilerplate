# Environment Variables Reference

Complete reference for all environment variables used in the SaaS Boilerplate.

## Platform Selection

These variables control which platform-specific implementations are used.

| Variable | Default | Options | Description |
|----------|---------|---------|-------------|
| `STORAGE_BACKEND` | `s3` | `s3`, `r2`, `b2`, `minio`, `local` | File storage backend |
| `EMAIL_BACKEND` | `django_ses.SESBackend` | See Email section | Email sending backend |
| `TASK_BACKEND` | `lambda` | `lambda`, `celery` | Background task execution |
| `TRACING_BACKEND` | `xray` | `xray`, `otel`, `none` | Distributed tracing |

**When to use which task backend:**

| Deployment | `TASK_BACKEND` | Why |
|------------|---------------|-----|
| **AWS (ECS/Lambda)** | `lambda` | Uses EventBridge + Lambda for serverless task execution |
| **Render.com** | `celery` | Uses Celery workers with Redis |
| **VPS (Docker)** | `celery` | Uses Celery workers with Redis |
| **Local Dev** | `celery` | Uses Docker Celery container |

## Core Django Settings

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `DJANGO_SECRET_KEY` | ✅ | - | Django secret key for cryptographic signing |
| `DJANGO_DEBUG` | | `False` | Enable debug mode (never in production) |
| `DJANGO_ALLOWED_HOSTS` | ✅ | `[]` | Comma-separated list of allowed hosts |
| `DJANGO_LOG_LEVEL` | | `INFO` | Logging level |
| `ENVIRONMENT_NAME` | | `` | Environment name (local, staging, production) |
| `CSRF_TRUSTED_ORIGINS` | | `[]` | Trusted origins for CSRF |
| `HASHID_FIELD_SALT` | ✅ | - | Salt for hashid field encoding |

**Generate secrets:**

```bash
# Generate DJANGO_SECRET_KEY
python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

# Generate HASHID_FIELD_SALT
openssl rand -hex 32
```

**URL formatting rules:**

| Variable | Format | Example |
|----------|--------|---------|
| `DJANGO_ALLOWED_HOSTS` | Hostnames only, no protocol | `api.example.com,backend.onrender.com` |
| `CSRF_TRUSTED_ORIGINS` | Full URLs with protocol | `https://example.com,https://www.example.com` |
| `WEB_APP_URL` / `API_URL` | Full URL with protocol | `https://api.example.com` |

| Variable | Required | Format |
|----------|----------|--------|
| `DB_CONNECTION` | ✅ | JSON: `{"host":"...","port":5432,"dbname":"...","username":"...","password":"..."}` |
| `DB_PROXY_ENDPOINT` | | RDS Proxy endpoint if using connection pooling |

### Connection Settings

| Variable | Default | Description |
|----------|---------|-------------|
| `DB_CONN_MAX_AGE` | `60` | How long to keep database connections open (seconds). **Critical for Render.com**: Set to `0` to close connections after each request and prevent connection exhaustion on limited plans |
| `HEALTH_CHECK_CACHE_TTL` | `30` | Cache duration for health check migrations query (seconds). Reduces database load from frequent health checks |

:::warning Render.com Connection Management
For Render.com with `basic-1gb` PostgreSQL (22-25 max connections):
- Set `DB_CONN_MAX_AGE=0` for backend-api to close connections after each request
- This prevents "too many connections" errors when multiple services share the database
- The connection math: Backend (2 workers) + Celery Worker (2 processes) + Celery Beat (1) + Headroom (3) = ~8-10 connections
:::

## CORS (Cross-Origin Resource Sharing)

:::warning Critical for Render.com and Other Split-Domain Deployments
When your frontend and backend are on different domains (e.g., Render.com), you **must** configure CORS. Without it, authentication will fail!
:::

| Variable | Format | Description |
|----------|--------|-------------|
| `CORS_ALLOWED_ORIGINS` | Full URLs with `https://` | Origins allowed to make cross-origin requests |
| `CSRF_TRUSTED_ORIGINS` | Full URLs with `https://` | Origins trusted for CSRF protection |

**When is CORS needed?**

| Deployment Type | Frontend URL | Backend URL | CORS Required? |
|-----------------|--------------|-------------|----------------|
| **Render.com** | `https://webapp-xxx.onrender.com` | `https://backend-xxx.onrender.com` | ✅ Yes |
| **VPS (Traefik)** | `https://example.com` | `https://api.example.com` | ❌ No (same parent domain) |
| **AWS (CloudFront)** | `https://app.example.com` | `https://api.example.com` | ❌ No (same parent domain) |

**Example for Render.com:**

```bash
# Both must include https:// prefix
CORS_ALLOWED_ORIGINS=https://webapp-abc123.onrender.com
CSRF_TRUSTED_ORIGINS=https://webapp-abc123.onrender.com,https://backend-abc123.onrender.com
```

## Celery Workers (when using `TASK_BACKEND=celery`)

| Variable | Default | Description |
|----------|---------|-------------|
| `CELERY_WORKER_MAX_TASKS_PER_CHILD` | `50` | Number of tasks a worker executes before restarting. Lower = more memory cleanup, higher = less overhead. Recommended: 50 for memory-constrained environments |

:::tip Celery Memory Management
For Render.com or other memory-constrained environments:
- `CELERY_WORKER_MAX_TASKS_PER_CHILD=50` restarts workers after 50 tasks to release memory
- Database connections are automatically closed after each task
- `CELERY_WORKER_PREFETCH_MULTIPLIER=1` (hardcoded) prevents task prefetching
:::

## Cookie Settings (Cross-Origin Authentication)

:::danger Critical for Render.com
These settings are **required** when frontend and backend are on different domains. Without them, login will fail because cookies won't be sent with cross-origin requests!
:::

| Variable | Default | Options | Description |
|----------|---------|---------|-------------|
| `COOKIE_SAMESITE` | `Lax` | `Lax`, `Strict`, `None` | Cookie SameSite attribute |
| `COOKIE_SECURE` | `True` (prod) | `True`, `False` | Require HTTPS for cookies |

**When to use which value:**

| Deployment | `COOKIE_SAMESITE` | `COOKIE_SECURE` | Why |
|------------|-------------------|-----------------|-----|
| **Render.com** | `None` | `True` | Cross-origin: different subdomains |
| **VPS (Traefik)** | `Lax` | `True` | Same parent domain |
| **AWS (CloudFront)** | `Lax` | `True` | Same parent domain |
| **Local Dev** | `Lax` | `False` | HTTP localhost |

## MCP Server Configuration

The MCP (Model Context Protocol) Server is an optional service that enables AI assistants to interact with your GraphQL API. It exposes GraphQL operations as MCP tools that AI models can use.

### MCP Server Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `GRAPHQL_ENDPOINT` | `http://backend:5001/api/graphql/` | Backend GraphQL endpoint URL |
| `MCP_LOG_LEVEL` | `info` | Log level: `debug`, `info`, `warn`, `error` |
| `MUTATION_MODE` | `explicit` | Mutation permissions: `none`, `explicit`, `all` |

### Mutation Modes

| Mode | Description | Use Case |
|------|-------------|----------|
| `none` | No mutations allowed | Read-only AI access |
| `explicit` | Only pre-defined mutations in `/operations` | **Recommended** - Controlled write access |
| `all` | AI can build any GraphQL mutation | Full access (use with caution) |

### AWS Infrastructure Variables (CDK)

When deploying to AWS with AI features enabled:

| Variable | Description |
|----------|-------------|
| `SB_AI_ENABLED` | Set to `true` to enable MCP Server deployment |
| `SB_MCP_SERVER_URL` | URL where MCP server will be accessible |
| `SB_DOMAIN_MCP` | Domain for MCP server (e.g., `mcp.staging.example.com`) |

### AWS (Backend Secrets)

For the backend container, add `OPENAI_API_KEY` via the backend secrets editor:

```bash
pnpm saas backend secrets
```

Add `"OPENAI_API_KEY": "sk-..."` to the JSON. The backend loads it from SSM at runtime via Chamber.

### AWS (MCP Server Secrets)

For the MCP server container, set env vars via:

```bash
pnpm saas mcp-server secrets
```

Add `OPENAI_API_KEY` and any other MCP-specific variables. The MCP container loads these from SSM at runtime via Chamber.

### Platform-Specific MCP Configuration

| Platform | MCP Server URL | Notes |
|----------|---------------|-------|
| **Local Dev** | `http://mcp-server:4000` | Via Docker Compose |
| **Render.com** | `https://mcp-server-xxx.onrender.com` | Separate service |
| **VPS** | `http://mcp-server:4000` | Internal Docker network |
| **AWS** | `https://mcp.{stage}.{domain}` | ECS Fargate service |

## Example Configurations

### Development (Minimal)

```bash
DJANGO_SECRET_KEY=dev-secret-key
HASHID_FIELD_SALT=dev-salt
DATABASE_URL=postgresql://backend:backend@db:5432/backend
REDIS_CONNECTION=redis://redis:6379

EMAIL_BACKEND=django.core.mail.backends.console.EmailBackend
TASK_BACKEND=celery
TRACING_BACKEND=none
STRIPE_CHECKS_ENABLED=False
```

### Production (Render.com)

See `.env.render.example` files in `packages/backend/` and `packages/webapp/` for complete examples.

```bash

HASHID_FIELD_SALT=<generated>

# DATABASE_URL=<from-render>
# REDIS_CONNECTION=<from-render>

# Platform backends
STORAGE_BACKEND=r2
EMAIL_BACKEND=sendgrid_backend.SendgridBackend

CORS_ALLOWED_ORIGINS=https://webapp-xxx.onrender.com
```
