# Step 3: Create Environment Stage

This step initializes your deployment environment (e.g., `qa`, `staging`, `production`) in AWS. You'll bootstrap the CDK and set configuration variables that define how your environment will be deployed.

**Estimated time: 20-30 minutes**

---

## What You'll Accomplish

- ✅ Bootstrap AWS CDK in your account
- ✅ Set deployment configuration variables
- ✅ Configure domain subdomains for your services
- ✅ Prepare AWS for infrastructure deployment

---

## Understanding Environment Stages

<ProjectName /> supports multiple isolated environments, each with its own:
- Database
- Application instances
- Configuration
- Domain subdomains

| Stage | Purpose | Example Domain |
|-------|---------|----------------|
| `qa` | Development/testing | `app.qa.example.com` |
| `staging` | Pre-production testing | `app.staging.example.com` |
| `production` | Live production | `app.example.com` or `app.production.example.com` |

:::tip Start with QA
For your first deployment, we recommend creating a `qa` environment. This lets you learn the process before deploying production.
:::

---

## Step 3.1: Bootstrap CDK

AWS CDK needs to prepare your account with resources it uses for deployments (S3 buckets, IAM roles, etc.). This only needs to be done once per AWS account/region.

```bash
# Make sure you're in the project root directory
pnpm saas infra bootstrap
```

This command:
- Creates CDK bootstrap resources in your AWS account
- Creates a KMS key for encrypting secrets (used by Chamber)
- Sets up the foundation for all future deployments

**Expected output:**
```
✅  Environment aws://123456789012/eu-west-1 bootstrapped.
```

:::info Already Bootstrapped?
If you see a message saying bootstrap was already run, that's fine. Each AWS account only needs to be bootstrapped once per region.
:::

---

## Step 3.2: Set Your Environment Context

Switch to the environment stage you want to configure:

```bash
# Replace [STAGE_NAME] with your stage (e.g., qa, staging, production)
pnpm saas aws set-env qa
```

This command:
- Sets environment variables for the current terminal session
- Configures subsequent commands to target the specified environment
- If using aws-vault, prompts for credentials/MFA

:::caution Terminal Session
This environment context only applies to the current terminal. If you open a new terminal, run this command again.
:::

---

## Step 3.3: Set Required Configuration Variables

Now configure the environment variables that define your deployment. These are stored securely in AWS Systems Manager Parameter Store.

### Project Name

Set the project name (should match `.env` file):

```bash
pnpm saas aws set-var PROJECT_NAME saas
```

:::tip Project Name Guidelines
- Use lowercase letters only
- Keep it short (used as prefix for AWS resources)
- No spaces or special characters
- Example: `saas`, `myapp`, `acme`
:::

### Domain Configuration

Use the Hosted Zone details from Step 2:

```bash
# Replace with your actual values
pnpm saas aws set-var SB_HOSTED_ZONE_ID Z0123456789ABCDEFGHIJ
pnpm saas aws set-var SB_HOSTED_ZONE_NAME example.com
```

Now set the domains for each service. Replace `example.com` and `qa` with your domain and stage name:

```bash
# Web application
pnpm saas aws set-var SB_DOMAIN_WEB_APP app.qa.example.com

# Backend API
pnpm saas aws set-var SB_DOMAIN_API api.qa.example.com

# Admin panel
pnpm saas aws set-var SB_DOMAIN_ADMIN_PANEL admin.qa.example.com

# CDN for static assets
pnpm saas aws set-var SB_DOMAIN_CDN cdn.qa.example.com

# Documentation (if deploying docs)
pnpm saas aws set-var SB_DOMAIN_DOCS docs.qa.example.com

# Celery Flower monitoring (optional)
pnpm saas aws set-var SB_DOMAIN_FLOWER flower.qa.example.com
```

## Step 3.4: Configure Optional Settings

### For Production Environments

If deploying production with root domain (e.g., `app.example.com` instead of `app.production.example.com`):

```bash
pnpm saas aws set-var SB_CERTIFICATE_DOMAIN example.com
```

Without this, you'll get a certificate mismatch error during deployment.

### Basic Auth Protection

Protect non-production environments with HTTP basic auth:

```bash
# Format: username:password
pnpm saas aws set-var SB_BASIC_AUTH demo:secretpassword123
```

:::tip Security
Always set basic auth for staging/QA environments to prevent unauthorized access.
:::

### CI/CD Mode

For new AWS accounts with CodeBuild limits:

```bash
# Use 'simple' for new accounts, 'parallel' for established accounts
pnpm saas aws set-var SB_CI_MODE simple
```

| Mode | Description | Recommended For |
|------|-------------|-----------------|
| `parallel` | Runs CI/CD stages concurrently | Established AWS accounts |
| `simple` | Runs CI/CD stages sequentially | New AWS accounts (avoids build limits) |

### Version Matrix (Optional)

If you want to track deployed versions across environments:

```bash
pnpm saas aws set-var SB_TOOLS_ENABLED true
pnpm saas aws set-var SB_TOOLS_BASIC_AUTH admin:password
pnpm saas aws set-var SB_TOOLS_HOSTED_ZONE_ID Z0123456789ABCDEFGHIJ
pnpm saas aws set-var SB_TOOLS_HOSTED_ZONE_NAME example.com
pnpm saas aws set-var SB_TOOLS_DOMAIN_VERSION_MATRIX status.example.com
```

See [Version Matrix](/working-with-sb/dev-tools/version-matrix) for more details.

## Complete Configuration Example

Here's a complete example for a `qa` environment with domain `example.com`:

```bash
# Set environment context
pnpm saas aws set-env qa

# Required: Project name
pnpm saas aws set-var PROJECT_NAME saas

# Required: Domain configuration
pnpm saas aws set-var SB_HOSTED_ZONE_ID Z0123456789ABCDEFGHIJ
pnpm saas aws set-var SB_HOSTED_ZONE_NAME example.com
pnpm saas aws set-var SB_DOMAIN_WEB_APP app.qa.example.com
pnpm saas aws set-var SB_DOMAIN_API api.qa.example.com
pnpm saas aws set-var SB_DOMAIN_ADMIN_PANEL admin.qa.example.com
pnpm saas aws set-var SB_DOMAIN_CDN cdn.qa.example.com
pnpm saas aws set-var SB_DOMAIN_DOCS docs.qa.example.com
pnpm saas aws set-var SB_DOMAIN_FLOWER flower.qa.example.com

# Recommended: Protect non-production environments
pnpm saas aws set-var SB_BASIC_AUTH demo:secretpassword

# Optional: For new AWS accounts
pnpm saas aws set-var SB_CI_MODE simple
```

| Variable | Required | Description | Example |
|----------|----------|-------------|---------|

## How Variables Are Stored

All variables are stored in AWS Systems Manager Parameter Store:
- Encrypted at rest using AWS KMS
- Accessible only to your AWS account
- Organized by environment: `/env-saas-qa-*/VARIABLE_NAME`

You can view them in the AWS Console under **Systems Manager → Parameter Store**.

## Verification

Confirm your configuration:

```bash
# Verify environment is set
pnpm saas aws get-env
# Expected: qa (or your stage name)

# View set variables in AWS Console
pnpm saas aws login
# Navigate to: Systems Manager → Parameter Store
# Filter by: /env-saas-qa
```

Check that you have parameters for:
- [ ] `PROJECT_NAME`
- [ ] `SB_HOSTED_ZONE_ID`
- [ ] `SB_HOSTED_ZONE_NAME`
- [ ] `SB_DOMAIN_WEB_APP`
- [ ] `SB_DOMAIN_API`
- [ ] `SB_DOMAIN_ADMIN_PANEL`
- [ ] `SB_DOMAIN_CDN`

## Troubleshooting

### "chamber: command not found"

Chamber CLI is required. Install it:
```bash
brew install chamber  # macOS
```

### "Unable to locate credentials"

Run `pnpm saas aws set-env [STAGE_NAME]` first to set AWS credentials.

### Variables not appearing in Parameter Store

- Ensure you ran `pnpm saas aws set-env [STAGE_NAME]` before setting variables
- Variables are stored under the path `/env-[PROJECT_NAME]-[STAGE_NAME]/`
- Check you're looking at the correct AWS region
