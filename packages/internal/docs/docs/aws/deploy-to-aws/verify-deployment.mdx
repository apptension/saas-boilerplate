## Deployment Verification Checklist

Work through each section to confirm your deployment is successful.

### 1. Web Application

**URL:** `https://app.qa.example.com` (replace with your domain)

- [ ] Page loads without errors
- [ ] Login form is displayed
- [ ] If basic auth is enabled, authentication prompt appears

**Test actions:**
1. Create a new user account
2. Verify the signup process completes
3. Login with the new account
4. Check that the dashboard loads

### 2. Backend API

**URL:** `https://api.qa.example.com/api/healthcheck/`

- [ ] Returns `{"status": "ok"}` or similar success response
- [ ] No 502/503 errors

**GraphQL Endpoint:** `https://api.qa.example.com/graphql/`

- [ ] GraphQL playground loads (if debug mode is enabled)
- [ ] Or returns appropriate response

### 3. Admin Panel

**URL:** `https://admin.qa.example.com/admin`

- [ ] Django admin login page loads
- [ ] Can login with `ADMIN_EMAIL` and `ADMIN_DEFAULT_PASSWORD`
- [ ] Django admin dashboard is accessible

**Immediate actions:**
1. Change the admin password
2. Verify user model shows registered users

### 4. Celery Workers

**Check via Flower:** `https://flower.qa.example.com`

- [ ] Flower login works (use `FLOWER_BASIC_AUTH` credentials)
- [ ] At least one worker is listed
- [ ] Worker shows as online/connected

**Alternative: Check ECS**
1. Go to AWS Console → ECS → Clusters
2. Find your cluster
3. Check that Celery services are running

### 5. Database

Verify via admin panel or API:
- [ ] Can create new records (e.g., user signup works)
- [ ] Data persists after refresh
- [ ] No database connection errors in logs

### 6. Email (If Configured)

- [ ] Password reset emails send successfully
- [ ] Account verification emails arrive

:::info SES Sandbox
If using a new AWS account, SES is in sandbox mode. You can only send to verified email addresses until you request production access.
:::

## Viewing Logs

If something isn't working, check the logs:

### CloudWatch Logs

```bash
pnpm saas aws login
```

Navigate to **CloudWatch → Log groups**:

| Log Group | Contains |
|-----------|----------|
| `/ecs/saas-qa-api` | Backend API logs |
| `/ecs/saas-qa-celery-default` | Celery worker logs |
| `/ecs/saas-qa-celery-beat` | Celery scheduler logs |
| `/aws/lambda/saas-qa-*` | Lambda function logs |

### SSH into Running Container

For debugging, you can access a running backend container:

```bash
pnpm saas backend remote-shell
```

This opens a bash shell inside the container where you can:
- Run Django management commands
- Check environment variables
- Debug issues interactively

## Common Post-Deployment Tasks

### Update Application Code

After making code changes:

```bash
pnpm saas aws set-env qa
pnpm saas build
pnpm saas deploy
```

Or push to your CI/CD-enabled branch for automatic deployment.

### Update Environment Variables

```bash
pnpm saas aws set-env qa
pnpm saas backend secrets  # Edit secrets
pnpm saas backend deploy api  # Redeploy to pick up changes
```

### Scale Resources

Adjust instance sizes in CDK code:
- `packages/infra/infra-shared/src/stacks/db/stack.ts` - Database size
- `packages/infra/infra-shared/src/stacks/main/stack.ts` - ECS task sizes

Then redeploy infrastructure:
```bash
pnpm saas infra deploy
```

### Delete an Environment

To remove an environment and all its resources:

```bash
pnpm saas aws set-env qa

# Delete in reverse order
pnpm saas infra destroy components
pnpm saas infra destroy ci
pnpm saas infra destroy db
pnpm saas infra destroy main
```

:::warning Destructive Operation
This permanently deletes all data including the database. Make backups first if needed.
:::
