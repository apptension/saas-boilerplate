# Step 1: Configure AWS Credentials

This step sets up secure AWS access using aws-vault. You'll create an IAM role for deployments and configure your local environment to use it safely.

**Estimated time: 20-30 minutes**

---

## What You'll Accomplish

- ✅ Store AWS credentials securely (not in plain text)
- ✅ Create an IAM role for deployments
- ✅ Configure your project to use the credentials
- ✅ Test that AWS access works

---

## Option A: Using aws-vault (Recommended)

aws-vault stores credentials encrypted in your system keychain. This is the recommended approach.

### Step 1.1: Get Your AWS Access Keys

1. Sign in to the [AWS Console](https://console.aws.amazon.com/)
2. Click your username in the top-right corner
3. Select **Security credentials**
4. Under **Access keys**, click **Create access key**
5. Select **Command Line Interface (CLI)**
6. Check the confirmation checkbox and click **Next**
7. Click **Create access key**
8. **Important:** Copy both the Access Key ID and Secret Access Key now. You won't be able to see the secret again.

:::warning Keep Your Credentials Safe
Never commit AWS credentials to git or share them publicly. Treat them like passwords.
:::

### Step 1.2: Add Credentials to aws-vault

```bash
# Create a new profile (you'll be prompted for the access key and secret)
aws-vault add my-profile
```

Enter your Access Key ID and Secret Access Key when prompted.

### Step 1.3: Create an IAM Role for Deployments

Instead of using your root credentials for everything, create a role specifically for <ProjectName /> deployments.

**In the AWS Console:**

1. Go to **IAM** → **Roles** → **Create role**
2. Select **AWS account** as the trusted entity type
3. Select **This account** (your own account)
4. Click **Next**
5. Search for and attach the **AdministratorAccess** policy
6. Click **Next**
7. Name the role: `SaaSBoilerplateAdminRole`
8. Click **Create role**

:::info Why a Separate Role?
Using an assumed role instead of your user credentials:
- Provides temporary credentials (more secure)
- Makes it clear which permissions are needed for deployment
- Allows easy revocation without affecting your main account
:::

### Step 1.4: Update the Role's Trust Policy

After creating the role:

1. Go to **IAM** → **Roles** → **SaaSBoilerplateAdminRole**
2. Click the **Trust relationships** tab
3. Click **Edit trust policy**
4. Replace with the following (update `<AWS_ACCOUNT_ID>` with your 12-digit account ID):

```json
{
    "Version": "2012-10-17",
    "Statement": [
        {
            "Effect": "Allow",
            "Principal": {
                "AWS": "arn:aws:iam::<AWS_ACCOUNT_ID>:root"
            },
            "Action": "sts:AssumeRole"
        }
    ]
}
```

5. Click **Update policy**

:::tip Finding Your Account ID
Your AWS Account ID is visible in the top-right corner of the AWS Console (click your username) or run `aws sts get-caller-identity`.
:::

### Step 1.5: Configure the AWS Profile

Now link the role to your aws-vault profile. Edit `~/.aws/config`:

```ini title="~/.aws/config"
[profile my-profile]
region = eu-west-1

[profile saas]
source_profile = my-profile
role_arn = arn:aws:iam::<AWS_ACCOUNT_ID>:role/SaaSBoilerplateAdminRole
region = eu-west-1
```

Replace:
- `<AWS_ACCOUNT_ID>` with your 12-digit AWS account ID
- `eu-west-1` with your preferred AWS region

**Common AWS Regions:**
| Region | Location | Use When |
|--------|----------|----------|
| `us-east-1` | N. Virginia | Best for US-based users |
| `us-west-2` | Oregon | US West Coast |
| `eu-west-1` | Ireland | European users |
| `eu-central-1` | Frankfurt | EU data residency requirements |
| `ap-southeast-1` | Singapore | Asia-Pacific users |

### Step 1.6: Configure Your Project

Update the `.env` file in your project root:

```dotenv title=".env"
PROJECT_NAME=saas
AWS_VAULT_PROFILE=saas
```

- `PROJECT_NAME`: Used as a prefix for all AWS resources (keep it short, lowercase, no special characters)
- `AWS_VAULT_PROFILE`: Must match the profile name in `~/.aws/config`

### Step 1.7: Test AWS Access

Verify everything is configured correctly:

```bash
# This should prompt you for your aws-vault password/keychain access
pnpm saas aws set-env local

# Test AWS access by listing S3 buckets (it's okay if the list is empty)
aws-vault exec saas -- aws s3 ls

# Open AWS Console in your browser (confirms credentials work)
pnpm saas aws login
```

If the commands complete without error, you're ready to proceed!

## Option B: Without aws-vault

If you prefer not to use aws-vault, you can store credentials directly (less secure).

### Direct Configuration

1. Run `aws configure` and enter your credentials
2. Set the `AWS_PROFILE` environment variable instead of using aws-vault

```bash
aws configure --profile saas
# Enter your Access Key ID, Secret Access Key, and region

# Then export before running commands:
export AWS_PROFILE=saas
```

:::warning Security Note
This stores credentials in plain text at `~/.aws/credentials`. Consider this approach only for development or testing environments.
:::

## Optional: Multi-Factor Authentication (MFA)

For enhanced security, especially in production, configure MFA.

### Why You Might Need MFA

Some AWS organizations require MFA for sensitive operations. You'll know you need it if you see errors like:

```
AccessDenied. User doesn't have permission to call iam:GetRole
```

### Configure MFA with aws-vault

1. In AWS Console, go to **IAM** → **Users** → **Your username** → **Security credentials**
2. Under **Multi-factor authentication (MFA)**, click **Assign MFA device**
3. Follow the setup wizard for your authenticator app
4. Copy the **MFA device ARN** (looks like `arn:aws:iam::123456789012:mfa/username`)

5. Add it to your `~/.aws/config`:

```ini title="~/.aws/config" {4}
[profile saas]
source_profile = my-profile
role_arn = arn:aws:iam::<AWS_ACCOUNT_ID>:role/SaaSBoilerplateAdminRole
mfa_serial = arn:aws:iam::<AWS_ACCOUNT_ID>:mfa/your-username
region = eu-west-1
```

Now aws-vault will prompt for your MFA code when needed.

## Troubleshooting

### "Unable to locate credentials"

- Ensure aws-vault is installed and `my-profile` exists
- Run `aws-vault list` to see configured profiles
- Make sure `source_profile` in the `[profile saas]` section matches your base profile name

### "Access Denied" errors

- Verify the role ARN is correct
- Check the trust policy allows your account to assume the role
- Ensure the role has `AdministratorAccess` policy attached

### aws-vault keychain/password prompts

- This is normal—aws-vault uses your system keychain for security
- On macOS, you may need to approve access in System Preferences → Security & Privacy
