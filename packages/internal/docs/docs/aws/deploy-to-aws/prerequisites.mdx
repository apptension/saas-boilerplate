## 1. AWS Account

You need an AWS account with administrator access. If you're part of an organization, ask your AWS administrator for credentials.

### Creating a New AWS Account

If you don't have an AWS account:

1. Go to [aws.amazon.com](https://aws.amazon.com/)
2. Click **Create an AWS Account**
3. Follow the signup process (requires credit card)
4. After creation, sign in to the [AWS Console](https://console.aws.amazon.com/)

:::caution New AWS Accounts Have Limits
Fresh AWS accounts have service limits that may affect deployment:

- **SES Sandbox Mode**: Can only send emails to verified addresses
- **CodeBuild**: Limited to 1 concurrent build (affects CI/CD speed)
- **CloudFront**: May require account verification

These limits typically increase automatically with account age and usage. See our [Troubleshooting Guide](/aws/troubleshooting) for workarounds.
:::

### Required Permissions

Your AWS user needs administrator permissions. If using a limited account, ensure you have access to:

- IAM (to create roles)
- ECS, ECR
- RDS, ElastiCache
- S3, CloudFront
- Route 53, ACM
- Secrets Manager, Systems Manager
- CodePipeline, CodeBuild
- CloudWatch, Lambda

## 2. Docker Hub Account

Docker Hub is required for pulling base images during builds. Without it, AWS CodeBuild will hit rate limits and fail.

1. Go to [hub.docker.com](https://hub.docker.com/)
2. Create a free account
3. Note your username and password (you'll need these later)

:::tip Why Docker Hub?
AWS CodeBuild shares IP addresses with other AWS customers. Without Docker Hub authentication, you'll quickly hit Docker's anonymous pull rate limits.
:::

</Tabs>

## 4. Install aws-vault

aws-vault securely stores AWS credentials and makes working with multiple AWS accounts easier. It's optional but **strongly recommended** for security.

<Tabs>
<TabItem value="mac" label="macOS">

```bash
# Using Homebrew
brew install aws-vault

# Verify installation
aws-vault --version
```

</TabItem>
<TabItem value="linux" label="Linux">

```bash
# Download the latest release
sudo curl -L -o /usr/local/bin/aws-vault https://github.com/99designs/aws-vault/releases/latest/download/aws-vault-linux-amd64
sudo chmod 755 /usr/local/bin/aws-vault

# Verify installation
aws-vault --version
```

</TabItem>
<TabItem value="windows" label="Windows">

```powershell
# Using Chocolatey
choco install aws-vault

# Or download from GitHub releases:
# https://github.com/99designs/aws-vault/releases
```

</TabItem>
</Tabs>

**Expected output:** `v7.x.x` (version number)

:::info Why aws-vault?
Storing AWS credentials in plain text (`~/.aws/credentials`) is a security risk. aws-vault encrypts your credentials using your operating system's keychain/credential manager.
:::

## 5. Install Chamber

Chamber manages environment variables in AWS Systems Manager Parameter Store. It's required for setting deployment configuration and secrets.

<Tabs>
<TabItem value="mac" label="macOS">

```bash
# Using Homebrew
brew install chamber

# Verify installation
chamber version
```

</TabItem>
<TabItem value="linux" label="Linux">

</TabItem>
<TabItem value="windows" label="Windows">

Download from [Chamber releases](https://github.com/segmentio/chamber/releases) and add to your PATH.

</Tabs>

## 7. Domain Name

For production deployments, you need a domain name. You have several options:

| Option | Best For | Notes |
|--------|----------|-------|
| **Buy through Route 53** | Simplest setup | AWS handles everything automatically |
| **Transfer existing domain** | Already own a domain | Requires DNS migration |
| **Use external DNS** | Complex requirements | More manual configuration required |

For testing, you can use AWS-generated subdomains, but production requires a real domain.

:::tip For Learning
If you're just testing the deployment process, you can proceed without a custom domain initially. The deployment will still work, but some features may be limited.
:::
